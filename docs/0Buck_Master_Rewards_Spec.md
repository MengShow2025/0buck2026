# 0Buck 奖励与分销系统全规格说明书 (Master Rewards Spec)

## 1. 系统概览
0Buck 奖励系统旨在通过高粘性（签到）、高爆发（拼团）和长尾收益（粉丝分销）构建增长闭环。系统包含六大核心支柱，并由一套**优先级引擎**驱动结算。

---

## 2. 六大核心支柱

### 支柱 1：签到返现 (Sign-in Cashback)
*   **触发条件**：订单确认收货且过退货期。
*   **周期**：500天内。
*   **结构**：分为 20 期发放，总额等于"奖励基数"（实付金额 - 运费 - 税费）。
*   **多订单处理 (Batch Check-in v3.4.7)**：
    *   用户每日只需执行一次“注入”操作，即可为当天所有活跃订单完成签到。
    *   每个订单独立核算进度、周期与奖励。
    *   前端智能展示最近达成里程碑/结算周期的订单。
*   **算法**：
    *   **P1-P2 (引导期)**：固定（4%, 6%），周期短（5d, 10d）。
    *   **P3-P20 (长期期)**：每期 25-30 天。包含常规期（3-5%）与爆点大奖期（8%, 10%）。
        *   **爆点出现机制（概率加权抽取）**：不再做绝对的期数隔离。P3-P20 从剩下的 18 种组合中全量抽取。
            *   **前期 (P3-P10)**：8% 和 10% 的爆点组合仅有 **20%** 的概率落入此阶段（更难出现，但有惊喜）。
            *   **后期 (P11-P20)**：8% 和 10% 的爆点组合有 **80%** 的概率落入此阶段（更容易出现，刺激坚持）。
        *   **权威算法参考（JavaScript）**：
            ```javascript
            /**
             * 0Buck 船长精简版生成器
             * 核心：P1-P2固定，P3-P20线性决定是否触发爆点池
             */
            const _ = require('lodash');

            function generate0BuckFinalRoadmap() {
                // 1. 初始 20 个组合全家桶
                const p1_p2 = [{ r: 4, d: 5 }, { r: 6, d: 10 }];
                const big_hits = [{ r: 8, d: 30 }, { r: 10, d: 30 }];
                const normal_30d = [
                    { r: 5, d: 30 }, { r: 5, d: 30 }, { r: 5, d: 30 }, { r: 5, d: 30 }, { r: 5, d: 30 }
                ];
                const normal_25d = [
                    { r: 5, d: 25 }, { r: 5, d: 25 }, { r: 5, d: 25 }, { r: 5, d: 25 },
                    { r: 3, d: 25 }, { r: 4, d: 25 }, { r: 4, d: 25 }, { r: 4, d: 25 },
                    { r: 4, d: 25 }, { r: 4, d: 25 }, { r: 4, d: 25 }
                ];

                // 2. 核心：全量抽取与概率加权
                let slots = new Array(18).fill(null);
                
                big_hits.forEach(hit => {
                    // 20% 概率落在 P3-P10(前8个), 80% 概率落在 P11-P20(后10个)
                    let target_stage = Math.random() < 0.8 ? 'late' : 'early';
                    let placed = false;
                    while (!placed) {
                        let idx = target_stage === 'early' ? 
                                  Math.floor(Math.random() * 8) : 
                                  Math.floor(Math.random() * 10) + 8;
                        if (slots[idx] === null) {
                            slots[idx] = hit;
                            placed = true;
                        }
                    }
                });

                let all_normals = _.shuffle([...normal_30d, ...normal_25d]);
                for (let i = 0; i < 18; i++) {
                    if (slots[i] === null) {
                        slots[i] = all_normals.pop();
                    }
                }

                // 3. 最终流水线
                return [...p1_p2, ...slots].map((item, index) => ({
                    phase: index + 1,
                    ratio: item.r,
                    days: item.d
                }));
            }
            ```
*   **断签规则**：
    *   任何一期中断，本期奖励作废。
    *   下次签到从**下一期**的第一天开始。
*   **续签机制**：每期可使用 1 次"续签卡"（积分兑换），最大断签天数需 ≤ 5天。

### 支柱 2：拼团购 (Group Buy / Share-to-Free)
*   **物理入口隔离**：
    *   此拼团分享入口 **极其隐蔽**，仅存在于用户**“已购买订单”**内部的特定商品。外部人员无法在商城公共区域获取此链接。
*   **触发条件**：受邀者通过该已购订单内的特定商品分享链接，产生新的付款订单。
*   **免单规则 (Progressive Settle Rule)**：
    *   **阶梯结算**：当用户购买了 N 件商品并发起拼团，每成功邀请 **3** 人购买，系统**立即结算 1 件商品免单**。
    *   **持续累加**：若继续邀请满下一个 3 人（累计 6 人），则发放第 2 件免单，依此类推。
    *   **封顶**：最多免单至购买数量 N 件。
*   **抉择期与 P1 冲突**：
    *   仅在 **第一期签到 (Phase 1 / 5天内)** 且未领取返现前可选。
    *   一旦领取 P1 返现或进入 P2，拼团免单资格永久失效。
*   **优先级**：低于“分销奖”，高于“推荐奖励/粉丝奖”。

### 支柱 3：用户分销 (User Direct Distribution)
*   **物理入口隔离**：全站所有商品详情页、商家主页均有公共的分享按钮，用户点击后生成包含其身份参数的“专属商品链接”。
*   **触发条件**：受邀者点击该链接并在当次会话中产生购买行为。
*   **奖励额度 (v3.4.8)**：
    *   **Silver (3.0%)** / **Gold (4.0%)** / **Platinum (5.0%)**。
*   **物理隔离优势**：由于该链接与已购订单的“拼团链接”在结构与入口上完全分离，系统无需在此处判断与拼团的冲突优先级，天然隔离。

### 支柱 4：用户推荐奖励 (User Fan Referral)
*   **物理入口隔离**：通过平台生成的 **“专属注册链接/海报”**（Registration Link）。
*   **机制**：建立 **2 年 LTV** 粉丝绑定关系（后台记录 `inviter_id`）。
*   **触发条件**：该粉丝在 2 年内产生的 **非分销专属链接** 订单（即自己在商城里逛着买的东西）。
*   **防腐机制**：2 年（730 天）后粉丝关系自动解绑。

### 支柱 5 & 6：达人分销与达人推荐奖励 (KOL System)
*   **身份一致性**：达人与用户逻辑完全相同，仅比例不同。
*   **准入与谈判**：达人需申请入住，比例由平台根据谈判结果最终达成并记录在 `dist_rate` 和 `fan_rate` 字段。
*   **默认比例范围**：
    *   **达人分销**：8% - 20% (Negotiated)。
    *   **达人推荐奖励**：3% - 8% (Negotiated)。

## 3. 优先级引擎逻辑 (Priority Engine)
当一个订单涉及多个推荐关系或满足多个奖励条件时，系统按以下优先级 **仅发放一种** 奖励，且各档比例随用户/达人等级动态调整：

1.  **达人分销 (KOL Distribution)**: 8% - 20%（优先级最高）。
2.  **用户分销 (User Direct Distribution)**: 3% - 5%（优先级次之）。
3.  **拼团免单 (Group Buy)**: 3人免单奖励（由用户主动选择触发）。
4.  **达人粉丝奖励 (KOL Fan Rewards)**: 3% - 8%（2年LTV）。
5.  **用户粉丝奖励 (User Fan Rewards)**: 1.5% - 3%（2年LTV）。

*注：*
*   *同一笔订单只产生一种分销/推荐奖励。*
*   *用户自身的"签到返现"是基础权益，除非用户选择了"拼团免单"，此时二者互斥。*

## 4. 结算与提现规则 (Settlement & Payout Rules)

### 4.1 核心原则 (Post-Return Period Settlement)
为确保财务安全，0Buck 遵循 **“有效完成订单 (Signed + 15D)”** 的严格结算原则：
*   **启动时机**：订单进入“有效完成”状态后的 **第二天**，系统才正式开启该订单的 500 天签到计划。
*   **异常挂起**：若发生退货、换货或物流纠纷，签到计划立即 **“停止并等待 (Suspend & Wait)”**，直到重新签收并度过新的退货期后，方可继续进度。
*   **奖励冻结**：
    *   **非交易积分**：行为完成即时发放（受日上限约束）。
    *   **交易类积分/奖励**：产生时即进入 `frozen` 状态，仅在订单“有效完成”后解冻。
    *   **穿仓处理**：若退款时积分已消费导致余额不足，无需向用户追回（积分价值较低，以用户体验为主）。

### 4.2 自动追回机制 (Clawback Logic)
若订单在有效完成前发生退款：
*   **资金扣除**：自动从受益人钱包中扣除 `pending` 状态的佣金与返现。
*   **状态作废**：该订单对应的计划立即转为 `forfeited`。

## 5. 达人 (KOL) 奖励深度逻辑

### 5.1 分销奖 (Distribution)
*   **逻辑**：纯链接驱动，不动（不分享/无成交）则无收益。

### 5.2 推荐奖励 (Referral / Fan Bond)
*   **有效期**：**严格 2 年期 (LTV)**。
*   **淘汰机制**：粉丝绑定关系在 2 年后自动解除，变回“路人”。
*   **增长压力**：达人与用户必须持续补充新用户进入网络，以抵消 2 年期满后的粉丝流失。

## 6. 积分系统与续签卡 (Points & Renewal Card)

### 6.1 积分获取规则 (Points Earning)

#### 6.1.1 非交易类奖励 (Daily Cap: 150 Pts)
为保持系统平衡，所有非交易行为产生的积分每日累计上限为 **150 Pts**：
*   **涵盖种类**：正常签到、写帖子、写评论、参与活动、参与预售/众筹、提建议等。
*   **结算时机**：行为完成即时发放（受日上限约束）。

#### 6.1.2 交易类奖励 (No Cap)
 交易驱动的积分奖励无上限，但遵循 **“有效完成订单”** 原则：
 *   **结算前提**：必须满足“签收 + 过退货期”的条件。
 *   **奖励基数**：
     *   **自己交易**：`订单实付金额 * 5`。
     *   **粉丝交易**：`订单实付金额 * 2`。
 *   **特性**：仅销售类积分与订单的有效性强绑定，若发生退款则按比例追回。
 
 ### 6.2 续签卡 (Renewal Card) 规则
 *   **兑换价格**：**3000 Pts / 张**。
 *   **使用限制**：每期仅限使用 1 次，断签后 5 天内必须使用。
 
 ### 6.3 Shopify 原生卡券系统集成 (Discount Code & Gift Card)
 针对用户的积压积分与客诉情况，系统深度整合 Shopify 原生的折扣码 (Discount Code) 系统，拓展积分消耗与情绪抚慰场景：
 *   **积分兑换体系**：用户可使用积分享受“动态面值优惠券”或“全场免邮券”。系统通过后台 Shopify API 实时生成一张绑定该用户邮箱的专属折扣码（如 `0B-PTS-5USD`）。
 *   **AI 抚慰发券 (The Peacemaker AI)**：AI 管家在监测到用户针对物流延迟、商品瑕疵等产生负面情绪时，可主动在聊天中生成并赠送 Shopify 抵用券。此举利用极低开发成本，将“被动补偿”转化为“主动维护”，大幅降低客服介入成本及平台争议风险。

 ## 7. 提现、财务与收入 (Withdrawal & Revenue)

### 7.1 提现手续费 (Withdrawal Fees)
*   **虚拟货币 (USDC/USDT)**：**$2 / 每次** (固定费用，**平台强烈鼓励使用**，手续费最低)。
*   **其他方式 (如 PayPal)**：**3.2% + $2 / 每次**。
*   **扣除方式**：所有手续费均直接从用户的 **提现金额内扣除**。
*   **最小提现金额**：**$50 USD**。

### 7.2 汇率管理逻辑 (Exchange Rate Management)
系统遵循三阶段汇率换算原则，以确保财务安全与成本对冲：

1.  **采购入库阶段 (CNY $\rightarrow$ USD)**：
    *   从 1688 等平台抓取商品时，原始人民币 (CNY) 价格进入服务器时立即按系统汇率换算为 **美元 (USD) 成本**。
    *   此阶段应用 **+0.5% 的汇率对冲 Buffer**。

2.  **前端展示阶段 (USD $\rightarrow$ Local Currency)**：
    *   服务器中的 USD 价格根据用户的地理位置/偏好币种，实时换算为 **本地货币 (如 EUR, GBP, JPY)** 展示给用户。

3.  **支付结算阶段 (Local $\rightarrow$ USD Payment)**：
    *   用户以本地货币金额支付时，支付网关/系统将其换算回 **美元 (USD) 支付金额** 进行最终财务结算。

### 7.3 平台收入
*   **来源**：汇率对冲产生的 Buffer (0.5%)、未兑付的准备金、以及提现手续费差价（若有）均归为 **平台收入**。

## 10. 零信任安全架构 (v3.5.0)

### 10.1 身份认证与访问控制 (Auth & AuthZ)
*   **JWT 核心化**：废弃基于 Cookie 的 Session 模拟，全站 API 强制校验由后端签发的加密 JWT。
*   **强制 IDOR 防护**：所有涉及个人数据的接口（钱包、订单、计划）强制执行 **“资源所有权校验”**，禁止通过修改 URL 中的 ID 越权访问他人数据。

### 10.2 逻辑安全防护 (Business Logic Security)
*   **时区跳跃防御**：签到逻辑统一采用 **UTC 零点** 作为结算基准，限制用户修改时区的频率（每 30 天仅限一次）。
*   **AI 安全沙箱**：Dumbo AI 的工具调用链中增加权限拦截层，严禁 AI 跨用户查询敏感内存或财务事实。

### 10.3 财务数据完整性 (Financial Integrity)
*   **复式记账对账**：系统每日凌晨自动执行 **“日终对账脚本”**。
    *   校验逻辑：`当前余额 == 初始余额 + Σ(收入流水) - Σ(支出流水)`。
    *   异常处理：若对账不平（Diff != 0），系统立即自动锁定相关账户提现功能并触发管理员报警。
*   **行级悲观锁**：涉及资金扣减的操作强制使用 `.with_for_update()`，彻底杜绝高并发下的“双花”漏洞。

## 8. 支付引擎与余额支付架构 (Payment Engine & Wallet Integration v3.5.0)
 
 为了确保 0Buck 的强金融属性与安全性，系统采用 **Shopify 原生 Discount Code API (方案 A)** 作为处理余额支付与抵扣的核心合规路径，彻底摒弃存在风控与举证风险的“改价式草稿订单”。
 
 ### 8.1 支付路径选择逻辑 (The Decision Tree)
 
 1.  **部分余额抵扣 (Partial Balance Payment)** $\rightarrow$ **方案 A (Discount Code API)**
     *   用户在 0Buck 前端选择使用部分余额（例如 $30）。
     *   后端即时调用 Shopify API，生成一张限定该用户邮箱、面值 $30 的专属一次性折扣码（如 `0BUCK-BAL-30USD`）。
     *   结账时自动应用该折扣码，用户跳转至 Shopify 收银台支付剩余现金。
     *   **合规优势**：若发生信用卡拒付（Chargeback）纠纷，Shopify 账单能清晰举证商品的原始价值（$100）及抵扣明细（$30 抵用券），而非被 Shopify 风控系统误判为恶意改价。
 2.  **全额余额支付 (Full Balance Payment)** $\rightarrow$ **方案 A (100% Discount Code)**
     *   用户余额足以覆盖订单全额（$0 现金支出）。
     *   后端生成等额面值的专属折扣码，实现 100% 抵扣。
     *   **优势**：通过合规折扣码生成的 0 元订单，完美规避了 Stripe/PayPal 的最低交易额限制，同时不会触发 Shopify “绕过支付网关”的风控警报，实现财务对账 100% 平账。
 
 ### 8.2 核心安全机制：价格防火墙 (Price Firewall)

*   **原则**：**绝对禁止信任前端传来的商品价格、运费或折扣金额。**
*   **执行流**：
    1.  前端仅发送 `product_id` 和 `quantity`。
    2.  后端根据数据库 `Product.sale_price` 重新计算总价。
    3.  后端计算 0.5% 的汇率 Buffer。
    4.  后端生成 Draft Order，确保支付金额由服务器端锁定。

### 8.3 余额“冻结-核销”闭环 (Freeze-Settle Cycle)

为了防止“双花”攻击（Double-Spending），系统执行以下金融原子操作：

1.  **冻结 (Freeze)**：
    *   用户点击“去支付”时，后端通过 `.with_for_update()` 锁定钱包。
    *   将待支付金额从 `balance_available` 移动到 `balance_locked`。
    *   记录 `WalletTransaction` 状态为 `pending_payment`。
2.  **核销 (Settle)**：
    *   当 Webhook 收到 `orders/paid` 或 Draft Order 支付完成信号时。
    *   将 `balance_locked` 中的对应金额正式扣除（销账）。
    *   更新 `WalletTransaction` 状态为 `completed`。
3.  **释放 (Release)**：
    *   若用户取消订单或 Draft Order 超时未支付（默认 2 小时）。
    *   将 `balance_locked` 中的金额退回到 `balance_available`。
    *   标记 `WalletTransaction` 为 `cancelled`。

### 8.4 余额支付的返现规则 (Rebate on Rebate)

*   **原则**：**允许使用奖励金额进行再购物，并产生新的返现。**
*   **返现基数计算**：
    *   新订单的返现基数（`reward_base`）= **订单总金额（含余额支付部分） - 运费 - 税费**。
    *   *注：余额支付部分被视为等同于现金的有效投入。*

## 11. 智慧商业与深度防御 (Smart Commerce & Deep Defense v3.6.0)

本章节定义了系统如何利用 AI 与动态路由技术，在将 Shopify “工具人化”的基础上实现利润最大化与极致用户体验。

### 11.1 Dumbo AI 弃单自动挽回 (AI Nudge Engine)

*   **监控目标**：Schema B (Draft Order) 生成后 1 小时内未支付的订单。
*   **挽回逻辑**：
    1.  系统每小时扫描一次待支付草稿订单。
    2.  Dumbo AI 根据用户历史偏好生成个性化话术。
    3.  通过 WhatsApp 发送包含 `invoice_url` 的提醒。
*   **业务目标**：将弃单转化率提升 15%-20%。

### 11.2 身份统一映射 (Zero-ID Mapping)

*   **核心理念**：用户在 0Buck 注册即视为在全平台注册。
*   **实现方式**：
    1.  创建订单时，后端自动检查/创建 Shopify Customer。
    2.  将 0Buck 的 `referral_code` 和 `user_tier` 同步至 Shopify Customer Tags。
    3.  **价值**：确保无论未来切换何种支付网关，用户资产与推荐关系链路始终在 0Buck 手中。

### 11.3 虚拟多仓库动态路由 (Virtual Sourcing Routing)

*   **运作机制**：
    1.  在创建 Draft Order 的瞬间，后端实时查询 1688 多个备选供应商的价格与库存。
    2.  自动选择 **“总成本最低 + 发货最稳”** 的货源。
    3.  将供应商 ID 注入订单元数据，供后续自动化采购使用。

### 11.4 汇率动态对冲 (Dynamic FX Buffer)

*   **策略**：
    1.  监控每日汇率波动率。
    2.  若波动率 < 0.2%，保持 0.5% Buffer。
    3.  若波动率 > 0.5%，自动提升至 0.8%-1.0% 以覆盖结算风险。

### 11.5 安全补丁：Webhook HMAC 强制验证

*   **加固项**：**生产环境严禁跳过 HMAC 校验。**
*   **逻辑**：所有来自 Shopify 的 Webhook 必须通过 `X-Shopify-Hmac-SHA256` 签名校验，否则直接拒绝请求，防止伪造订单支付信号。

## 9. 数据库模型扩展 (Database Extensions)
*   **Referrals表**：记录 `inviter_id`, `invitee_id`, `expiry_date`, `type (USER/KOL)`。
*   **RewardPhases表**：记录每个订单的 20 期执行状态。
*   **Points表**：记录用户积分，用于兑换"续签卡"。
*   **UserLevels表**：记录用户/达人等级及对应奖励系数，支持后台动态配置。
